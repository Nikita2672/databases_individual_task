В качестве прототипа реализуется календарное приложение с поддержкой оффлайн-режима и последующей синхронизацией, на котором апробируются мои гипотезы касаемо модели конфликтов в распределенных системах.

- Понимая, что сама тематика CRDT относится преимущественно к распределённым системам, предлагаю связать её с содержанием курса следующим образом, сделав индивидуальное задание по следующей теме:

- Проектирование и анализ схемы хранения композитных объектов, представляемых на основе pure operation-based CRDT, в системе управления базами данных.

- В рамках работы предполагается: – разработка схемы хранения операций, а также векторных часов; – анализ нормализации и целостности данных; – проектирование индексов и анализ типовых запросов; – сравнение возможных вариантов хранения (реляционная модель, например PostgreSQL, и документная модель, например MongoDB). На текущем этапе исследования окончательный выбор типа СУБД (реляционная или документная) ещё не зафиксирован, поскольку рассматривается гипотеза о возможных преимуществах хранения операций в JSON-представлении. ## PostgreSQL vs MongoDB

В рамках выполненного обзора были рассмотрены два принципиально разных подхода к хранению одного и того же библиотечного контракта `Operation`.

### Единица хранения

В PostgreSQL основной единицей хранения является отдельная операция. Журнал операций представлен как множество независимых строк таблицы `operations`, связанных с объектами и репликами через идентификаторы.

В MongoDB основной единицей хранения является документ объекта. История операций не выносится в отдельную коллекцию, а хранится как встроенный массив `operations` внутри документа `objects`.

### Сильные стороны PostgreSQL

- строгая и прозрачная схема хранения;
- явная ссылочная целостность;
- хорошая поддержка append-only operation log как множества независимых записей;
- эффективные точечные и аналитические запросы по `op_id`, `tx_id`, `object_id`;
- удобный анализ поведения запросов через `EXPLAIN ANALYZE`.

### Сильные стороны MongoDB

- естественное хранение полуструктурированных полей `clock` и `action`;
- удобное чтение полной истории конкретного объекта из одного документа;
- близость структуры хранения к прикладному объектному представлению;
- корректная работа multikey-индексов по вложенным операциям.

### Ограничения PostgreSQL

- часть структуры операции остается внутри `JSONB`, поэтому схема является лишь частично нормализованной;
- широкие низкоселективные выборки могут переходить в `Seq Scan`;
- сортировка длинной истории объекта может становиться заметной частью стоимости запроса.

### Ограничения MongoDB

- размер документа ограничен `16 MB`;
- очень длинная история одного объекта делает embedded-модель менее устойчивой;
- глобальные выборки по операциям через весь датасет менее естественны, чем в PostgreSQL;
- при дальнейшем росте истории объекта может потребоваться сегментация документа, а не хранение всей истории в одном массиве.

### Итоговый вывод

PostgreSQL оказался более естественным выбором для хранения глобального immutable operation log и для аналитических сценариев, где операция рассматривается как самостоятельная единица данных.

MongoDB оказался более естественным выбором для объектно-центричных сценариев, где важно быстро получить целиком историю конкретного объекта в форме, близкой к прикладному представлению.

Следовательно, различие между PostgreSQL и MongoDB в данном задании сводится не только к сравнению двух СУБД, но и к сравнению двух разных моделей хранения:

- PostgreSQL моделирует данные как журнал независимых операций;
- MongoDB моделирует данные как объектный агрегат со встроенной историей.
